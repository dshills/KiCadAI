# KiCad File Writers Implementation Plan

## Overview

This plan implements the project, schematic, and printed circuit board writer specification in small phases. The immediate goal is to generate a new KiCad project from Go without depending on live KiCad write support. The first complete acceptance target is a deterministic LED indicator project containing:

```text
led_indicator.kicad_pro
led_indicator.kicad_sch
led_indicator.kicad_pcb
```

Each phase should be implemented, tested, reviewed, and committed before starting the next phase.

## Baseline Decisions

These decisions unblock Phase 1. They can be revised only through a spec update.

1. Writers remain under `internal/kicadfiles` until output compatibility is proven.
2. The first implementation targets KiCad 10-compatible output.
3. Schematic generation initially references KiCad standard libraries instead of embedding custom `.kicad_sym` libraries.
4. PCB generation initially references standard footprints and supports a minimal generated board model; full custom footprint generation is deferred.
5. Project JSON is modeled in Go, but implementation may start from a minimal KiCad-generated fixture to confirm required fields.
6. Integration validation is opt-in and must not use IPC write calls.

## Phase 1: S-Expression Writer

### Objective

Build the structured rendering foundation for `.kicad_sch` and `.kicad_pcb`.

### Tasks

1. Create package:

```text
internal/kicadfiles/sexpr
```

2. Implement structured node types for:
   - Atom.
   - Quoted string.
   - Integer.
   - Float.
   - Fixed-point numeric string.
   - List.
   - Optional omitted node.
3. Implement renderer with:
   - Stable indentation.
   - UTF-8 output.
   - `\n` line endings.
   - Final trailing newline.
   - String escaping for quotes, backslashes, and control characters.
   - Deterministic float formatting.
4. Add helper constructors so writer packages do not assemble raw strings.
5. Add tests:
   - Atom rendering.
   - String escaping.
   - Nested list formatting.
   - Float formatting.
   - Fixed-point numeric rendering.
   - Optional omission.
   - Golden schematic fragment.

### Deliverables

- `internal/kicadfiles/sexpr`.
- Unit tests and golden fragment.

### Done When

- `go test ./internal/kicadfiles/sexpr` passes.
- No schematic/PCB writer uses raw S-expression string concatenation.

## Phase 9: Cross-File Design Model and Validation

### Objective

Connect project, schematic, and PCB data through a single design model.

### Tasks

1. Add:

```go
type Design struct
```

2. Add cross-file validation:
   - Project name matches output basenames.
   - Schematic references map to PCB footprint references where provided.
   - Paired schematic symbols and PCB footprints share component logical paths.
   - PCB nets include expected schematic net names where mapping is supplied.
   - Board footprints have unique references.
   - UUIDs are unique across all generated schematic sheets and the PCB.
   - Hierarchical schematic sheet references form a directed acyclic graph.
3. Add fixture builder:

```go
func LEDIndicatorDesign(input LEDIndicatorInput) (Design, error)
```

4. Add tests:
   - Valid LED design.
   - Missing PCB net for schematic net.
   - Duplicate footprint reference.
   - Project/schematic name mismatch.

### Deliverables

- Cross-file design model.
- Cross-file validation.
- LED design builder.

### Done When

- Complete LED design can be produced in memory and validated without writing files.

## Phase 10: Safe Project Directory Writer

### Objective

Write complete generated designs to disk safely.

### Tasks

1. Implement:

```go
func WriteProjectDirectory(root string, design Design, opts WriteOptions) (WriteResult, error)
```

2. Add behavior:
   - Create project directory.
   - Refuse unsafe overwrite.
   - Treat any existing target directory as an overwrite case, even when it is empty.
   - Write files inside a dedicated operation temporary directory created with `os.MkdirTemp(targetParent, prefix)` so the temporary project and final target are on the same filesystem.
   - For new project directories, rename the completed temporary project directory into place in one operation only when the target does not exist and the temp directory is on the same filesystem.
   - Return a structured error without partial copying if the new-project directory rename fails.
   - Normalize generated project-relative paths to Unicode Normalization Form C before validation or persistence.
   - For explicit overwrites, create a unique random-suffix backup container with `os.MkdirTemp`, rename the existing project directory to a child path inside that container, rename the completed temporary project directory into place, remove the backup container after success, and report any backup cleanup warning in `WriteResult`.
   - Fail before moving the existing project if no unique backup container can be created.
   - Create and update an overwrite recovery journal before each directory swap phase.
   - Detect an existing recovery journal for the project and refuse to start a second swap until recovery is handled.
   - Attempt best-effort rollback by moving the backup child directory back to the target path if the final rename of the new project fails.
   - Return `WriteResult.BackupDir` and `WriteResult.JournalPath` when an overwrite swap fails after moving the existing project or when backup cleanup fails.
   - Remove the operation temporary directory on ordinary failures.
   - Omit PCB when nil.
   - Return written file list.
3. Add structured warnings.
4. Add tests:
   - Creates expected files.
   - Refuses existing non-empty directory.
   - Allows explicit overwrite.
   - Cleans temp files after induced failure.
   - Reports journal and backup paths after induced swap failure.
   - Simulates each overwrite journal phase and verifies recovery reporting.
   - Verifies best-effort rollback restores the original target after final rename failure.
   - Omits PCB when nil.

### Deliverables

- Safe file-system writer.
- Write result model.

### Done When

- New project directories can be published with a single rename, and overwrite swaps have explicit backup recovery reporting.
- Existing user files are not overwritten without explicit permission.

## Risk Handling

### KiCad Rejects Generated Schematic

Response:

1. Compare generated output with KiCad fixture.
2. Add a failing golden or integration test.
3. Update writer ordering/required fields.
4. Keep change narrowly scoped.

### Standard Library Symbol References Are Insufficient

Response:

1. Add project-local symbol fixture support.
2. Extend schematic writer to emit `lib_symbols`.
3. Keep custom symbol library writer as a separate spec if it grows.

### Standard Footprint References Are Insufficient

Response:

1. Add footprint fixture embedding or project-local footprint library support.
2. Defer arbitrary footprint writer until the fixture path works.

### Generated Files Differ After KiCad Resave

Response:

1. Identify whether the difference is canonical ordering or semantic repair.
2. Update writer to match canonical ordering for harmless diffs.
3. Treat semantic repair as a writer bug.

### Direct Writer Scope Expands Into Parser/Editor

Response:

1. Reject scope creep in the current implementation.
2. Add a future spec for parsing and lossless editing if needed.
